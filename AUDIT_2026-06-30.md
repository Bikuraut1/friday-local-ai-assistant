# FRIDAY Folder Audit & Improvement Plan

Audited: 2026-06-30 · Scope: `D:\Friday` (structure, hygiene, security, ops) · Auditor: Claude (Cowork)

This audit looks at how the folder is organized and operated, not whether the AI features work. The phase report already covers feature status well; this complements it. Overall the project is impressively built and documented — the issues below are about hygiene, reproducibility, and exposure, not architecture.

---

## Top priorities (do these first)

1. **Initialize real version control + add a `.gitignore`.** A `.git/` folder exists but is empty — there is no repository, no commit history, and no `.gitignore` anywhere. Eleven phases of work have zero history or rollback ability.
2. **Bind Docker ports to localhost only.** Every service publishes on all network interfaces (`0.0.0.0`), and Open WebUI has `WEBUI_AUTH=False`. Anyone on the same Wi-Fi/LAN can reach the chat UI, n8n, Qdrant, and SearXNG with no login.
3. **Stop tracking/bundling generated junk.** Three `.venv` folders (~70,000 files combined), the Qdrant vector store, `__pycache__`, and logs sit inside the project. This bloats the folder and would poison any future git repo or copy/move.

## Security findings

### S1 — Services exposed on all interfaces without auth (High)
All `docker-compose.yml` files publish ports as e.g. `"3000:8080"`, which Docker binds to `0.0.0.0` (every interface). Combined with `WEBUI_AUTH: "False"`, the following are reachable by any device on your network with no credentials: Open WebUI (3000), n8n (5678), n8n automation (8788), Qdrant (6333/6334), SearXNG (8081), Kokoro (8880).

Fix — prefix each published port with `127.0.0.1:` so it only listens locally:

```yaml
ports:
  - "127.0.0.1:3000:8080"
```

Apply the same to every compose file. If you ever do want LAN access, turn on `WEBUI_AUTH` first.

### S2 — Hardcoded secrets / weak defaults in compose files (Medium)
- `n8n/docker-compose.yml`: `FRIDAY_OPENWEBUI_PASSWORD: "admin"` (admin/admin default, in two places).
- `searxng/docker-compose.yml`: `SEARXNG_SECRET: "friday-local-searxng-secret"`.
- `open-webui/docker-compose.yml` and `rag/reranker.py`: shared `friday-local-*` API keys.

These are local-only values so the risk is limited today, but they become real once S1 is unfixed or the folder is shared. Move them into per-service `.env` files (git-ignored) and reference with `${VAR}` in compose. Change the `admin`/`admin` pair to a real password.

### S3 — Email credential handling is correct (Good)
`agent/tools/email_handler.py` reads SMTP user/password from environment variables, not from disk. Keep this pattern; extend it to the items in S2.

## Hygiene & structure findings

### H1 — No version control (High)
`.git/` is empty. Initialize a repo, commit the current state as a baseline, and commit per logical change going forward. This gives you rollback, diffs, and a safety net the project currently lacks.

### H2 — No `.gitignore` (High, blocks H1)
Without one, a `git add .` would commit ~70k venv files, the vector DB, caches, and logs. A ready-to-use file is included at the end of this audit — drop it in before the first commit.

### H3 — Dependency reproducibility is weak (Medium)
Only `mem0/requirements.txt` exists. `agent`, `voice`, `dashboard`, `rag`, `router`, `vision`, and `n8n/scripts` have running code but no pinned manifest. If a venv is lost you cannot reliably rebuild. Add a `requirements.txt` per Python service (`pip freeze` is a quick start), and a `package.json` for the n8n `.mjs` scripts.

### H4 — Data and code are intermingled (Medium)
The Qdrant store lives at `mem0/qdrant/storage/...` (many 32 MB mmap files). Application state inside the source tree makes backups, moves, and version control awkward. Consider a Docker named volume or a separate `D:\Friday-data\` path, leaving the project folder code-only.

### H5 — Generated output accumulates with no retention (Low)
`logs/` (44 files), `maintenance/reports/` (14), `n8n/output/` (timestamped briefings piling up), and `vision/screenshots/` all grow unbounded. Add a simple rotation/cleanup step (keep last N, delete older than X days) — a scheduled `maintenance` script would fit your existing pattern.

### H6 — Empty/placeholder directory (Low)
`agent/config/` is empty. Either populate it or remove it so the tree reflects reality.

### H7 — No top-level README (Low)
The entry point is `FRIDAY_PHASE_1_TO_11_REPORT.md`, written "for ChatGPT as a reviewer." Add a short `README.md` with a one-paragraph description, prerequisites, and the three operator commands (`start/status/stop`). Keep the phase report as the deep-dive.

## Operations & consistency

### O1 — Stack startup is fragmented (Medium)
Five separate compose files brought up via PowerShell. There is no single `docker compose up` for the whole stack. Consider a root `docker-compose.yml` (or a compose `include:`/override) so the Docker services come up together, with the PowerShell scripts handling only the native pieces (Ollama, voice, router, dashboard).

### O2 — Possible duplicate automation logic (Low)
`n8n/scripts/` has both `friday_automation.py` and `friday_automation.mjs` plus `automation_api.mjs`. If Python and Node are doing the same job, pick one to reduce drift; if they're intentionally different, a one-line comment at the top of each clarifying its role would help.

### O3 — Backups: good policy, gaps remain (Low)
Two ~9 MB backups exist and the backup script sensibly excludes huge vector/model stores. But: backups aren't scheduled yet (the report notes startup auto-launch isn't installed), and the vector store has no backup path at all. Schedule a weekly lightweight backup via Task Scheduler and a monthly `-IncludeVectorStores` run, then actually test a restore once.

## What's already done well

Consistent `friday-*` naming across containers and scripts; a per-module `README.md` in every subsystem; sensible model routing for an 8 GB-VRAM box; environment-variable email credentials; `restart: unless-stopped` on all containers; and a genuine maintenance layer (health report, backup, ops runbook). The documentation is well above typical for a personal project.
